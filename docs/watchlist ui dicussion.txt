Feeding a screenshot to a coding agent is a strong starting point, but relying on it exclusively has a major drawback for your stack: **image-to-code models default to generating raw HTML/CSS or Tailwind rather than idiomatic Ant Design components.** A better approach is a **Hybrid Strategy**: provide the screenshot for visual context, but explicitly define the Ant Design (`antd`) components, TypeScript interfaces, and hidden interactive states in your text prompt.

### Why Pure Screenshot Prompting Falls Short

* **Component Ignorance:** The agent might build a custom list or table from scratch instead of leveraging `antd`'s `<Table>` or `<List>`, forcing you to manually rebuild sorting, filtering, and theme consistency later.
* **Hidden States:** Screenshots capture static pixels. The agent will completely miss Kite's signature UI interaction—the Buy, Sell, and Market Depth buttons that only appear on row hover.

### 1. Left Side Panel: Navigation & Quick Edits

* **Component:** Use an `antd` `<Tree>` component with the `draggable` prop enabled. This is superior to using multiple `<Collapse>` panels because `<Tree>` supports native drag-and-drop for both parent nodes (watchlists) and leaf nodes (stocks) out of the box.
* **Expansion Logic:** Enable the `multiple` selection prop on the `<Tree>` to allow multiple watchlists to remain open simultaneously.
* **Inline Actions:** Utilize the `titleRender` prop to inject a custom React layout into each tree node. This will display the watchlist name alongside hidden `<Button type="text">` icons (Rename, Add, Remove) that only become visible via CSS `hover` states.
* **Stock Addition:** Clicking the "Add" icon will trigger an `antd` `<Select showSearch>` or `<AutoComplete>` input embedded directly below the node, querying your global instrument database.
* **Persistence:** The `<Tree onDrop>` event handler must capture the new positional indices and trigger an immediate `UPDATE` to your Supabase database, ensuring the custom layout loads exactly the same way on the next session.

### 2. Management Screen: The Master Data View

* **Data Model:** A centralized master view where each row represents a unique stock.
* **Component:** An `antd` `<Table>` with `rowSelection={{ type: 'checkbox' }}` enabled to facilitate bulk operations.
* **Columns:**
* **Symbol:** The core identifier.
* **Watchlist Memberships:** Displayed as a collection of `antd` `<Tag>` components (e.g., a stock can show tags for "Watchlist 1" and "Watchlist 2" simultaneously). Clicking a tag filters the grid.
* **Theme/Conviction Tags:** A dedicated column for custom thematic tagging, allowing you to label specific instruments as "Pick and Shovel" or "Coca-Cola vs Refrigerator" plays.
* **Priority Ranking:** An `antd` `<Rate count={5}>` component to visually assign and display conviction levels 1 through 5.
* **Description:** An `<Input.TextArea>` or a lightweight rich-text editor for inline tracking of your long-term thesis notes over the next 20 years.


* **Bulk Actions:** A sticky header bar that activates only when table rows are checked, exposing buttons to apply a Priority rank or append a Tag to all selected stocks at once.

### Guidance & Architectural Review

Visualizing a literal many-to-many node-link graph (like a web of bubbles) quickly becomes visually overwhelming and practically useless for managing data. The Master Data Grid (Table) approach described above is the industry standard for resolving many-to-many relationships cleanly. It flattens the complexity by making the stock the focal point and treating watchlists simply as attributes (tags) applied to that stock.

To make this architecture work without performance degradation, you must normalize your React state. Store a single dictionary of `stocksById`, a single dictionary of `watchlistsById`, and a mapping array that links them. This ensures that updating a stock's priority or description on the Management Screen instantly reflects inside the Left Side Panel without requiring the UI to iterate through and update duplicated objects.

### Dual-Access Note Tracking Strategy

To ensure you never lose the long-term logic—such as why a specific company is a crucial "pick and shovel" play or fits the "Coca-Cola vs. Refrigerator" model—you need an architecture that supports both a quick daily glance and a dedicated space for deep-dive writing.

**1. Main Watchlist Window: Quick View & Edit (The Slide-Out Drawer)**

* **The Trigger:** Inside your Left Side Panel (`<Tree>`), inject a subtle `antd` `<FileTextOutlined>` icon next to the symbol of any stock or watchlist that currently contains notes.
* **Quick Glance (Hover):** Wrap that icon in an `antd` `<Tooltip>`. On hover, display a plaintext preview (e.g., the first 100 characters) of your thesis to provide context without requiring a click.
* **Full Access (Click):** Clicking the icon triggers an `antd` `<Drawer>` component.
* **Configuration:** Set the Drawer `placement="right"` and `width={400}` so it slides out over your visualization space without completely obstructing the main interface.
* **Content:** Render the full description, the 1-5 Priority Rank, and the applied Theme Tags.
* **Inline Editing:** Embed an `antd` `<Form>` inside the Drawer. This allows you to update your conviction notes live while looking at market data, rather than forcing a context switch to another screen.



**2. Master Management Screen: Bulk Editing View**

* **The Editor:** On the Many-to-Many management `<Table>`, dedicate a specific column for "Thesis / Moat".
* **Component:** Use an `antd` `<Input.TextArea>` with the `autoSize` prop enabled for inline editing across multiple rows simultaneously.
* **State Synchronization:** Because the React state is normalized, updating a note here triggers an immediate sync to the global store, ensuring the text inside the Main Watchlist Drawer is always current.

**Architectural Guidance for Performance:**
Separate your data fetching. Do not include these heavy text descriptions in the high-frequency WebSockets or API polling used for the live `TickData`. Keep the price payload lean. Fetch the thesis data from Supabase only when the `<Drawer>` is explicitly opened or when the Master Management Screen mounts.
